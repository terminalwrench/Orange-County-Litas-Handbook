# Communication Workflow

Clear communication keeps operations from relying on memory, side chats, or assumptions.

## Communication Layers

| Layer | Purpose | Where to Document |
| --- | --- | --- |
| Public event information | What members and guests need to know. | Event posts, email, event records. |
| Internal planning | Ownership, tasks, timing, and decisions. | [../09-Leadership](../09-Leadership/README.md) |
| Canonical event record | What happened and what should be remembered. | [../10-Events-Database](../10-Events-Database/README.md) |
| Reusable knowledge | Lessons that apply beyond one event. | [../13-Knowledge](../13-Knowledge/README.md) |

## Internal Updates

Internal updates should include:

- What changed.
- Who owns the next step.
- What deadline matters.
- Whether the information is public yet.
- What decision is needed.

## Public Updates

Public updates should be:

- Clear.
- Timely.
- Consistent across channels.
- Specific about date, time, location, ride status, and weather changes.

If one public channel changes, update every place the old information appeared.

## Decision Documentation

Document decisions when they affect:

- Safety.
- Reputation.
- Partnerships.
- Money.
- Recurring operations.
- Leadership roles.
- Public commitments.

Use [../09-Leadership/Decision-Log.md](../09-Leadership/Decision-Log.md) for internal decision tracking.
